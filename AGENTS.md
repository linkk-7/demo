# 项目工作规约

本项目是双目视觉位移监测系统。当前代码库正在从“分步骤脚本 + 批处理”逐步整理为“模块化 + 少入口 + 面向在线处理”的结构。

## 总体原则

1. 先分析，再最小改造，不要一上来全面重写。
2. 一次只处理一个模块，尽量不要跨多个模块同时大改。
3. 优先保留已有核心算法实现，避免推翻重写。
4. 旧脚本先降级为兼容入口或 legacy 工具，不要立即删除。
5. 主流程目标是：采集一对图 -> 立即进入后续处理，而不是先批量存图再统一后处理。

## 当前主流程方向

目标主链路：

1. 采集一对左右图
2. 立体校正
3. 提取关键点 / 关键点 ID
4. 前后帧匹配 / 跟踪
5. 计算视差 / 三维 / 位移
6. 输出或发送结果

## 采集层规则（已基本稳定）

采集层位于 `get_dis_imgs/`，当前逻辑不要轻易推翻。

### 当前采集层已确定的规则

- 采集层输出 `FramePacket`
- 文件命名使用：
  - `L_{left_host_timestamp}_{frame_id:06d}.png`
  - `R_{right_host_timestamp}_{frame_id:06d}.png`
- `host_timestamp` 用于文件命名
- `dev_timestamp_raw` 保留在 packet 中作为底层原始元数据
- 左右图通过共同 `frame_id` 表示为一对
- retention 只负责最近 N 对图像缓存，不依赖 `frame_id`
- `frame_id` 采用有限范围循环编号（默认 `1~999999`）
- 启动时扫描缓存目录并避开已占用 `frame_id`
- 当前不生成统一 pair 主时间戳
- 当前不输出 `pair_delta_ms`
- 当前采集层详细行为优先阅读 `get_dis_imgs/README.md`

### 当前采集层主接口

- `capture_and_save_frame_pair(...)`
- `continuous_capture(...)`

## 已降级为 legacy / offline 的部分

以下内容不再属于当前主流程：

- `2.3_gain_t_and_rename_disp_images.py`
- `get_t.get_t()`
- `get_t.rename()`

这些只作为历史旧数据兼容或离线工具保留。

当前主流程不再依赖：
- 从文件名重新解析时间
- 将图片重命名成 `1l.jpg / 1r.jpg`

## 立体校正层当前结论

当前下一优先级是立体校正层。

### 已明确的判断

- `2.2.0_rectification.py` 本质上是旧批处理入口，应逐步降级为兼容脚本
- `2.2.1_val_rectification.py` 是校正结果验证脚本，不属于主流程
- `dist_and_stereo_recitify2/` 中应保留的是“单对图像立体校正核心能力”
- 新主流程应逐步支持：
  - 输入 `FramePacket`
  - 输出 `RectifiedPacket`（或等价结构）

### 立体校正层重构原则

处理该模块时请优先：
1. 区分核心算法与批处理胶水逻辑
2. 抽取“单对图像立体校正”能力
3. 先新增单对接口，再考虑旧入口降级
4. 不要一上来删除旧批处理代码

## 跟踪 / 位移计算层

当前暂未重构。
未经明确要求，不要提前大改。

## 工作方式要求

处理一个模块时，请遵循：

1. 先分析当前代码职责
2. 区分：
   - 核心能力
   - 批处理胶水逻辑
   - 验证/实验脚本
   - legacy 工具
3. 优先抽取单对/单帧能力
4. 修改后说明：
   - 改了哪些文件
   - 为什么这样改
   - 对当前主流程有什么意义
5. 除非明确要求，否则不要删除旧代码，优先降级为 legacy

3. 跟踪 / 关键点 ID 层
